# Implementation Plan: Session-Aware Navigation (Step 3.1)

## 1. Summary

Step 3 added `POST /login` / `GET /logout`, which correctly set and clear
`session["user_id"]` — but nothing in the UI ever reads that session
state. The navbar (`templates/base.html`, shared by every page since all
templates extend it) unconditionally shows "Sign in" / "Get started",
even after a successful login, so a logged-in user visiting `/` sees no
sign of being logged in and can click "Sign in" again. This step makes
the shared navbar session-aware: show a welcome message and a "Sign out"
link when logged in, and hide the "Sign in"/"Get started" links in that
case. Because the nav lives in `base.html`, this fix automatically
applies to every page (landing/home included), not just one route — no
per-page changes needed. `/profile` will pick this up for free once it
stops being a stub.

This is a follow-up correction to Step 3, not a new spec — no new
branch/spec, per user's request. Implement on the existing
`feature/03-login-and-logout` branch alongside the rest of Step 3.

## 2. Source

No separate spec — scoped directly from user feedback: after login, the
home page still offers "Sign in" and gives no logged-in feedback or
sign-out option.

## 3. Key Technical Facts

- Flask injects `session` into the Jinja template context automatically
  — no new route/context processor is needed to read
  `session.get('user_id')` inside `base.html`.
- `session` currently only stores `user_id` (set in `app.py`'s `login()`
  at the point `session["user_id"] = user["id"]`). There's no user name
  in the session, so a "Welcome, <name>" message needs the name added at
  login time too.

## 4. Implementation Order

1. `app.py` — store the user's name in the session at login time.
2. `templates/base.html` — make the navbar conditional on session state.

## 5. File-by-File Plan

### `app.py` — `login()` (the `POST` branch, where
`session["user_id"] = user["id"]` currently sits)

Add one more line right next to it: `session["user_name"] = user["name"]`.
Small, same spot, no control-flow change. (`logout()`'s
`session.pop("user_id", None)` should also pop `"user_name"` so nothing
stale survives a logout.)

### `templates/base.html` — `.nav-links` block (currently lines 21-24)

Wrap the existing two links in `{% if session.get('user_id') %}` /
`{% else %}` / `{% endif %}`:
- **Logged in branch**: a short "Welcome, {{ session.user_name }}" text
  node, plus a link to `url_for('logout')` labeled "Sign out" — same
  markup pattern as the existing anchors (reuse `.nav-cta`-style class for
  the sign-out link so it doesn't need new CSS).
- **Logged out branch**: the current "Sign in" (`url_for('login')`) and
  "Get started" (`url_for('register')`) links, unchanged.

No new CSS files needed — reuse existing `.nav-links`/`.nav-cta` classes
from `static/css/style.css` for the logged-in state's markup so it's
visually consistent with the current nav.

## 6. Edge Cases

- Visiting any page (not just `/`) while logged in must show the
  logged-in nav — this falls out for free since `base.html` is shared by
  every template.
- After `/logout`, the nav must revert to "Sign in"/"Get started" on the
  very next page load — covered by popping both `user_id` and
  `user_name` in `logout()`.
- A session with `user_id` but somehow no `user_name` (e.g. a session
  cookie created before this change) should degrade gracefully — use
  `session.get('user_name')` with a sensible fallback (e.g. "there") in
  the template rather than letting a missing key render blank/error.

## Verification

1. Log in with the seeded demo account, then visit `/` — expect a
   "Welcome, Demo User" message and a "Sign out" link in the nav, and no
   "Sign in"/"Get started" links.
2. Click "Sign out" from `/` — expect redirect to `/`, nav reverts to
   "Sign in"/"Get started".
3. Visit `/terms` or `/privacy` while logged in — expect the same
   logged-in nav (proves it's shared via `base.html`, not landing-page-
   specific).
4. Without logging in, visit `/` — expect the original "Sign in"/"Get
   started" nav (no regression for anonymous visitors).
